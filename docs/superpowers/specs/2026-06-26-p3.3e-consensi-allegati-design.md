# P3.3e — Blocchi Consensi & Allegati nel Form Builder + upload generico server-side

**Fase:** P3 (Amministrativo & finanziario) · **Slice:** P3.3e · **DL:** DL-029 · **Migrazione:** 20260746
**Branch:** feat/master-plan-registro

## Obiettivo

Completare i blocchi di contenuto del Form Builder (Sistema A: `form_models.schema`) con un
**blocco Consensi/Privacy** (cattura del consenso GDPR con evidenza legale) e la **rifinitura del
blocco Allegati** già esistente, introducendo un **endpoint di upload generico server-side** che
ripara l'upload allegati nel wizard autenticato (oggi rotto) e chiude il gate mancante sulla route
del builder.

## Contesto / stato reale (verificato live)

- **Due sistemi form coesistono:** Sistema A `form_models`(+`form_submissions`) = builder drag&drop,
  scoring, delibera, OTP (moderno); Sistema B `forms_templates`(+`forms_submissions`) = autorizzazioni
  legacy per classe. **Questa slice opera SOLO sul Sistema A.**
- **Allegati già funzionanti:** tipo `file` in palette + `FileField` + bucket privato `form_attachments`.
  L'upload pubblico iscrizione passa da `/api/iscrizione/upload` (service-role).
- **`storage.objects` ha ZERO policy RLS** → tutti i bucket sono service-role-only (deny-by-default).
  Conseguenza: nel wizard **autenticato** (`StepRenderer`) il `FileField` ricade sull'upload diretto
  via client browser (anon) → **DENY → rotto**.
- **`form_submissions` ha RLS** (`fs_insert_authenticated` ecc.) che richiede sessione Supabase Auth.
  L'app usa identità app-level (header `x-user-id`), quindi il client browser è anon → **anche
  l'insert non-firma del wizard è rotto**. Le submission vanno instradate su endpoint server-role.
- **Blocco Consensi: inesistente.** Nessun tipo `consent`, nessun riferimento consensi/privacy.
- **`/api/admin/form-models` è UNGATED** (usa `createAdminClient` senza `requireStaff`).
- `form_submissions.status` enum = `draft | pending_signature | completed`.

## Decisioni di prodotto (fissate con l'utente)

1. **Scope:** Consensi block + upload generico + sicurezza allegati nella stessa slice.
2. **Modello consenso:** **1 blocco = 1 consenso** (testo + link informativa facoltativo + toggle
   obbligatorio), reso come **una checkbox** da spuntare. Evidenza legale: in submission si registra
   il valore + **snapshot del testo accettato + timestamp** (`consents_log`).
3. **Sicurezza allegati:** **service-role + scoping app** (coerente con tutto l'app, decisione P0).
   Nessuna policy `storage.objects`. Letture staff via endpoint server-role scoped.

## Design

### 1. Tipo campo `consent` (Sistema A)
- `FormFieldType += 'consent'`. `FormField` guadagna optional: `text?` (corpo del consenso),
  `link?` (URL informativa), `link_label?` (etichetta link), `accept?`/`max_size_mb?` (per `file`).
- **Valore in submission:** `data[field.id] = true|false` (boolean). Richiesto ⇒ deve essere `true`.

### 2. Rendering — `FieldRenderer`
- `consent`: titolo (`label`) + corpo (`text`) + link opzionale + **checkbox** "Acconsento" (Controller,
  boolean). Validazione `required` ⇒ `validate: v => v === true || 'Devi accettare per proseguire'`.
- Il `consent` NON è un campo "decorativo": entra nella validazione e nello strip condizionale come
  un input normale (eccezione: i decorativi restano `section_header|paragraph|signature`).

### 3. Builder — palette + PropertiesPanel
- Palette: nuovo item **"Consensi/Privacy"** (`type:'consent'`, icona ShieldCheck). `makeField('consent')`
  default `required:true`.
- `PropertiesPanel` per `consent`: Etichetta (titolo), **Testo consenso** (textarea→`text`),
  **Link informativa** (`link`) + **Etichetta link** (`link_label`), toggle Obbligatorio. Niente
  scoring/opzioni per consent.
- `PropertiesPanel` per `file`: **Tipi file ammessi** (`accept`, es. `.pdf,.jpg`) + **Dimensione max MB**
  (`max_size_mb`).
- `campiDisponibili` (condizioni) esclude i consent dal poter essere *referenziati* solo se decorativi —
  i consent restano referenziabili (sono input boolean). *(Manteniamo l'esclusione attuale dei soli
  decorativi.)*

### 4. Helper puro — `src/lib/forms/consensi.ts` (cuore TDD)
- `estraiConsensi(pages: FormPage[], data, acceptedAt: string): ConsentoSnapshot[]`
  → per ogni campo `consent`: `{ field_id, label, text?, link?, accepted: boolean, accepted_at }`.
- `consensiObbligatoriMancanti(pages, data): string[]` → id dei consensi `required` non `=== true`
  (guard server-side difensiva; il wizard già blocca lato client).

### 5. Persistenza `consents_log`
- **Migrazione 20260746:** `ALTER TABLE form_submissions ADD COLUMN consents_log jsonb` (nullable).
- **`POST /api/forms/send-otp`** (path firma): dopo l'insert, carica `form_models.schema`, calcola
  `estraiConsensi` → salva `consents_log`; guard `consensiObbligatoriMancanti` ⇒ 400 se mancano.
- **`POST /api/forms/submit`** (NUOVO, path senza firma, service-role): sostituisce l'insert
  client-side rotto in `WizardContainer`. Crea submission `completed` con `consents_log`, stessa
  guard consensi + (riusa) `assertGenitoreNonSospeso`. Restituisce `{ id }`.

### 6. Upload generico — `POST /api/forms/upload` (NUOVO, service-role)
- Multipart `file` + `folder` (modelId). `requireUser` (autenticato qualsiasi) + **rate-limit** per IP.
- Valida dimensione (default 8MB, override `max_size_mb`) + tipo (whitelist estensioni/MIME). Salva in
  `form_attachments/models/{modelId}/{uuid}-{safeName}`, ritorna `{ path }`.
- **Wiring:** `StepRenderer` passa `uploadEndpoint="/api/forms/upload"` al `FieldRenderer` → ripara
  l'upload autenticato. (Public EnrollmentWizard resta su `/api/iscrizione/upload`.)
- *(Slice 2 "Pubblica modello" aggiungerà la variante pubblica token-scoped.)*

### 7. Lettura allegati staff — `GET /api/forms/attachment?path=` (NUOVO, service-role)
- `requireStaff` → restituisce **signed URL** (TTL breve) per il path richiesto, vincolato al bucket
  `form_attachments`. Read-side dello scoping app.

### 8. Gate route builder
- `POST`/`PATCH /api/admin/form-models` → anteporre `requireStaff(request)` (chiude il gap ungated).

## Boundary / cosa NON è in scope
- Niente policy `storage.objects` (decisione: service-role).
- Niente variante upload pubblica (slice 2). Niente modifiche al Sistema B (`forms_templates`).
- Niente solleciti/cron (regola utente).
