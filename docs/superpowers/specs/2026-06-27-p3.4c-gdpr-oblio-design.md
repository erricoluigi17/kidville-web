# P3.4c — GDPR Diritto all'oblio (anonimizzazione)

**Fase:** P3 · **Slice:** P3.4c · **DL:** DL-034 · **Migrazione:** 20260751
**Branch:** feat/master-plan-registro

## Obiettivo
Diritto all'oblio in due passi: **(1)** lista degli **alunni non iscritti** (+ genitori collegati)
candidati alla cancellazione; **(2)** dalla lista, **cancellazione definitiva = ANONIMIZZAZIONE**
(nessuna DELETE di righe), preservando **audit + registri fiscali**, con **dry-run + doppia conferma**.

## Decisioni di prodotto (fissate con l'utente)
1. **Flusso a 2 passi**: lista `alunni` con `stato <> 'iscritto'` (+ `parents` collegati via
   `student_parents`) → da lì si esegue la cancellazione definitiva del soggetto scelto.
2. **Solo anonimizzazione**: NESSUNA DELETE di righe (zero rischio FK). I campi PII vengono sovrascritti
   con placeholder `CANCELLATO-{hash}`; le righe operative restano e puntano all'anagrafica anonimizzata.
3. **Preserva audit + fisco (pseudonimizzati)**: `audit_scritture_docente`, `fascicolo_accessi_audit`,
   `sblocchi_audit`, `registro_modifiche` e `pagamenti`/`fatture_emesse` **non** si toccano come righe
   (obbligo legale di conservazione, GDPR art.17(3)(b)); risultano de-identificati perché l'anagrafica a
   cui puntano è anonimizzata. **Il bucket `fatture` NON viene purgato** (conservazione fiscale).
4. **Dry-run + doppia conferma**: anteprima conteggi senza scrivere; esecuzione solo con `confirm`
   uguale al nominativo del soggetto; gate **Direzione**; **log dell'oblio**.

## Contesto (verificato live)
- `alunni.stato` ∈ {`iscritto`} (128 oggi); candidati oblio = `stato <> 'iscritto'`.
- PII primaria: `alunni` (nome, cognome, codice_fiscale/fiscal_code, note_mediche, allergies/allergeni,
  residence_*, birth_*, documento_path, invoice_holder_details, intestatario_fatture) e `parents`
  (first_name, last_name, fiscal_code, emails, phone_numbers, residence_*, birth_*, document_number,
  documento_path, auth_user_id).
- Superficie FK: l'alunno è referenziato in ~20 tabelle operative (presenze, valutazioni, diario,
  certificati, pagelle, mensa, scrutini, documenti, legami…). **Restano** (no delete): de-identificate
  via l'anagrafica anonimizzata.
- File storage con PII: `alunni.documento_path` / `parents.documento_path` (bucket `form_attachments`/
  iscrizioni), `certificati_medici` (bucket `certificati-medici`), `student_documents`. **Vanno rimossi**
  (un binario non si anonimizza). `fatture` **escluso**.

## Design
### Migrazione 20260751
`ALTER TABLE alunni ADD COLUMN anonimizzato_il timestamptz;`
`ALTER TABLE parents ADD COLUMN anonimizzato_il timestamptz;`
(marca i soggetti cancellati → idempotenza + esclusione dalla lista).

### Helper puro — `src/lib/gdpr/anonimizza.ts` (cuore TDD)
- `placeholderFor(id)` → `CANCELLATO-${hash8(id)}` (deterministico).
- `patchAlunno(id)` → oggetto update PII per `alunni` (placeholder/null + `anonimizzato_il`).
- `patchParent(id)` → oggetto update PII per `parents`.
- `nomeConferma(alunno)` → stringa attesa per la doppia conferma (`COGNOME NOME`).
- `confermaValida(input, alunno)` → boolean (case/space-insensitive).

### `GET /api/admin/gdpr/candidates` (gate Direzione)
- Lista `alunni` con `stato <> 'iscritto'` e `anonimizzato_il IS NULL`, ciascuno con i `parents` collegati
  (via `student_parents`). Per ogni candidato: id, nome/cognome, classe, genitori (nome).

### `POST /api/admin/gdpr/erase` (gate Direzione, audit)
- body `{ alunno_id, mode: 'dryrun' | 'execute', confirm? }`.
- carica l'alunno (404 se assente). **Rifiuta** se `stato = 'iscritto'` (si cancella solo un non iscritto).
- **dryrun**: ritorna i conteggi per tabella/file che verrebbero anonimizzati/rimossi (alunno + parents
  orfani + n. file), **senza scrivere**.
- **execute**: richiede `confermaValida(confirm, alunno)` → altrimenti 400. Poi:
  1. anonimizza `alunni` (`patchAlunno`) + `anonimizzato_il=now`.
  2. per ogni `parent` collegato **orfano** (nessun altro alunno con `stato='iscritto'` e non anonimizzato):
     anonimizza `parents` (`patchParent`) + sgancia `auth_user_id` + `anonimizzato_il=now`.
  3. rimuove i **file storage** PII del soggetto (documento_path alunno/parents, file certificati_medici);
     **NON** tocca il bucket `fatture`.
  4. `logScrittura('gdpr_oblio', azione='update', valoreDopo={alunno_id, parents, files})`.
- ritorna il riepilogo eseguito.

### UI `/admin/gdpr` + `OblioPanel`
- Tabella candidati (alunno non iscritto + genitori). Azione **"Cancella definitivamente"** → modale:
  mostra l'**anteprima dry-run** (conteggi) + campo per **digitare il nominativo** (doppia conferma) →
  esegue. Gate Direzione lato server, fetch con `x-user-id`.

## Cosa NON è in scope
- DELETE di righe (solo anonimizzazione). Hard-delete fisico. Anonimizzazione del testo libero nelle note
  operative (si de-identifica via l'anagrafica). Anonimizzazione di account `utenti` staff. Purga del
  bucket `fatture` (conservazione fiscale).
