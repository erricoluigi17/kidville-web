# P3.4b — Multi-Sede CRUD (registry scuole)

**Fase:** P3 · **Slice:** P3.4b · **DL:** DL-033 · **Migrazione:** 20260750
**Branch:** feat/master-plan-registro

## Obiettivo
Gestione **Multi-Sede**: registry delle scuole/sedi con **aggiungi / rinomina / disattiva** e **config
isolata** per sede, riservata alla Direzione.

## Contesto (verificato live)
- **Non esiste** una tabella `scuole`/`schools`/`sedi`: la sede è un `scuola_id` **UUID hardcoded**
  (`11111111-1111-1111-1111-111111111111`) usato come soft-reference in `sections`/`utenti`/`alunni`
  (1 sola sede oggi).
- Esiste un pannello impostazioni admin (`/admin/impostazioni`) con tanti `*Settings` per-modulo, e
  `getModuleConfig` per le config per-modulo. Staff RBAC (DL-028) usa il gate Direzione
  `requireStaff(['admin','coordinator'])` + `/admin/staff` + `StaffPanel` — **pattern di riferimento**.

## Decisioni
1. **Registry `scuole`** (migr. 20260750): `id uuid pk`, `nome text not null`, `attiva boolean default
   true`, `citta text`, `indirizzo text`, `config jsonb default '{}'`, `created_at`, `updated_at`.
   **Seed** della sede esistente (`11111111-…`, nome "Kidville"). **Niente FK** su sections/utenti/alunni
   in questa slice (additivo e sicuro; `scuola_id` resta soft-reference — nota per slice future).
2. **Disattiva ≠ elimina**: la disattivazione è `attiva=false` (soft). L'eliminazione hard di una sede
   è fuori scope (pericolosa) → la gestirà semmai la slice GDPR.
3. **Gate Direzione** `requireStaff(['admin','coordinator'])` (coerente con Staff RBAC DL-028) + audit
   `logScrittura('multi_sede')`.
4. **Config isolata**: colonna `config jsonb` per-sede (editor key/value basico). Le config per-modulo
   esistenti restano keyed by `scuola_id` altrove.

## Design
### Migrazione 20260750
`CREATE TABLE scuole (...)` + seed `ON CONFLICT (id) DO NOTHING` + trigger `set_updated_at` se presente.
RLS: abilitata con policy service-role-permissiva? **No** — coerente con l'app (accesso via service-role
endpoint). Per evitare l'advisor `rls_enabled_no_policy` di tipo INFO, **non** abilito RLS su `scuole`
(come molte tabelle del progetto che restano senza RLS; accesso solo via service-role gated).

### Helper puro — `src/lib/scuole/validate.ts`
- `validaNomeScuola(nome)` → `{ ok, error? }` (non vuoto, ≤120 char).
- `normalizzaScuola(input)` → record sanificato (trim nome/citta/indirizzo).

### `/api/admin/schools` (gated `requireStaff(['admin','coordinator'])`)
- **GET** → lista scuole (ordinate per nome).
- **POST** `{ nome, citta?, indirizzo? }` → crea (valida nome) + audit `insert`.
- **PATCH** `{ id, nome?, citta?, indirizzo?, attiva?, config? }` → rinomina / disattiva / config + audit
  `update`. 404 se inesistente; valida nome se presente.

### UI `/admin/schools` + `SchoolsPanel`
- Lista sedi (nome, città, badge Attiva/Disattivata). Azioni: **Aggiungi** (form nome/città/indirizzo),
  **Rinomina** (inline), **Attiva/Disattiva** (toggle). Header gate Direzione (come `/admin/staff`),
  fetch con `x-user-id`.

## Cosa NON è in scope
- FK/migrazione dati su `scuola_id`. Hard-delete sede. Spostamento alunni tra sedi (P4). Editor
  avanzato delle config per-modulo per sede (resta il `config` jsonb generico).
